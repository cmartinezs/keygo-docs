# Phase 11: Feedback

## Overview

Feedback is the continuous process of collecting, analyzing, and acting on insights from users, team, and system metrics.

## Key Objectives

- [ ] Collect and analyze user feedback
- [ ] Run team retrospectives
- [ ] Document lessons learned
- [ ] Plan improvements and iterations
- [ ] Share insights across organization

## Files to Complete

### 1. **user-feedback.md**
**Purpose**: Collect and summarize user feedback

**Sections**:
- Feedback channels (surveys, interviews, support tickets)
- Feature requests and ideas
- Bug reports and pain points
- Usage patterns and insights
- NPS (Net Promoter Score) trends

**Analysis**:
- Frequency of mentions (high-impact issues first)
- Sentiment and urgency
- Linked to features/requirements

### 2. **retrospectives.md**
**Purpose**: Team reflection on what went well and what to improve

**Per retrospective** (sprint, release, or project):
- **Date**: When did we run it?
- **Team**: Who participated?
- **What Went Well**: Celebrate successes
- **What Could Improve**: Growth opportunities
- **Action Items**: Specific changes for next iteration
- **Metrics**: Velocity, quality, deployment frequency

**Example**:
```
## Sprint 5 Retrospective (Feb 1-15, 2024)

**What Went Well** ✅
- Strong code review discipline
- Smooth deployment
- Early user feedback

**What Could Improve** 🔄
- Testing took 40% longer than estimated
- Documentation lagged behind implementation

**Action Items**:
- [ ] Improve test automation (@alice)
- [ ] Document as we go (team policy)
```

### 3. **lessons-learned.md**
**Purpose**: Key insights from the project

**Per learning**:
- Context: What situation taught us this?
- Insight: What did we learn?
- Impact: How does it change our approach?
- For next time: How will we apply it?

**Examples**:
- Architectural choices that worked well or poorly
- Process improvements
- Technology selections (what to do/avoid)
- Team dynamics insights

---

## Continuous Feedback Loop

**Monthly**:
- Collect user feedback
- Review metrics and dashboards
- Share updates with team

**Per Sprint**:
- Sprint retrospective
- Update roadmap based on feedback
- Close feedback loop with users

**Per Release**:
- Release retrospective (what we shipped)
- Analyze impact (metrics, user adoption)
- Lessons learned

---

**Files**:
- `user-feedback.md`
- `retrospectives.md`
- `lessons-learned.md`

**Time Estimate**: 2-3 hours per cycle  
**Team**: Product, Engineering, all contributors  
**Output**: Continuous improvement and learning

**Frequency**: Monthly feedback, sprint retros, release reviews

---

## Tips

1. **Make feedback safe**: Create psychological safety for honest retrospectives
2. **Act on feedback**: If you collect it but don't act, you lose credibility
3. **Close the loop**: Tell users what you did with their feedback
4. **Celebrate wins**: Acknowledge what's working
5. **Learn from failures**: Failures are data, not blame
