## Retrospective Format

### Template

```markdown
# Sprint/Release Retrospective: [Name]

**Date**: [YYYY-MM-DD]
**Team**: [Team name]
**Participants**: [Who attended]

## What Went Well ✅
- Success 1
- Success 2

## What Could Improve 🔄
- Improvement area 1
- Improvement area 2

## Action Items
- [ ] Action 1 (Owner: @name, Due: date)
- [ ] Action 2 (Owner: @name, Due: date)

## Metrics
- Velocity: [points]
- Bug rate: [%]
- Deployment frequency: [x/week]
```

## Running Retrospectives

### Methods

| Method | Best For | Duration |
|--------|----------|----------|
| **Start/Stop/Continue** | Quick feedback | 30 min |
| **Mad/Sad/Glad** | Emotions and feelings | 45 min |
| **4Ls** | Liked, learned, lacked, longed for | 60 min |
| ** Sailboat** | Risk assessment | 60 min |

### Tips for Good Retros
1. **Create safety**: Non-judgmental environment
2. **Be specific**: Concrete examples
3. **Focus on action**: Not just discussion
4. **Follow up**: Track action items

## Action Items

### Action Item Template

```markdown
- [ ] [Specific action]
    - **Owner**: @name
    - **Due**: YYYY-MM-DD
    - **Status**: [Not Started | In Progress | Done]
```

### Action Item Categories

| Category | Examples |
|----------|----------|
| **Process** | Update standup time, improve code review |
| **Technical** | Add tests, refactor code |
| **Communication** | Better documentation, team updates |
| **Tools** | New linter, better monitoring |

## Example Retrospectives

### Example: Sprint Retrospective

# Sprint 12 Retrospective (March 1-14, 2024)

**Team**: Platform Team
**Participants**: 5 engineers, 1 PM

## What Went Well ✅
- Deployment to production with zero rollback
- Code review completed within 2 hours average
- New team member ramped up quickly
- Bug rate lower than previous sprint

## What Could Improve 🔄
- Testing took 40% longer than estimated
- Documentation lagged behind code
- Some unclear requirements from product

## Action Items
- [ ] Improve test automation (@alice, Due: March 20)
- [ ] Update documentation as we code (@bob, Due: ongoing)
- [ ] Clarify requirements before sprint planning (@pm, Due: March 18)

## Metrics
- Velocity: 34 points (baseline: 32)
- Bug escape: 2 bugs to production
- Deploy frequency: 3 times/week

### Example: Release Retrospective

# Release 2.0 Retrospective

**Date**: March 15, 2024
**Team**: Full engineering team

## What Went Well ✅
- SSO feature shipped on time
- Customer feedback very positive
- Smooth deployment process

## What Could Improve 🔄
- SSO required more work than estimated
- Documentation needs improvement
- Mobile app deprioritized too late

## Action Items
- [ ] Create better estimation for SSO features (@lead, Due: April)
- [ ] Improve documentation process (@docs, Due: March 30)
- [ ] Review roadmap earlier for mobile (@pm, Due: April)
